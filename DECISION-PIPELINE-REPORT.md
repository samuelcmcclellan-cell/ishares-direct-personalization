## 1. Decision Factors Ranked by Influence

The factors below are ranked from most influential (can override everything) to least influential (marginal score adjustments). "Influence" is measured by how much a single answer can move the final portfolio outcome.

### Tier A — Hard Overrides (bypass scoring entirely)

#### A1. Near-Term Capital Preservation Trigger
- **Source**: `matchingEngine.js` lines 175–180
- **Inputs**: Goal (from `goal` step) + Effective Timeline (from `timeline` step or `goal-followup` impliedTimeline)
- **Condition**: Goal ∈ {`home`, `education`, `emergency`} AND timeline = `under-2`
- **Effect**: Returns the `near-term-reserve` portfolio immediately. Risk score is still computed but **ignored** for selection. This is the only true hard override in the system.

#### A2. Preference Variant Overrides
- **Source**: `matchingEngine.js` lines 182–193
- **Inputs**: `preferences` toggles (income, intl, taxAware) and goal=income
- **Answers**: Three binary toggles on the `preferences` step; goal=`income` also triggers the income variant
- **Effect**: If `taxAware` is toggled → returns `tax-aware-balanced` portfolio. If `intl` → returns `global-explorer`. If `income` toggle OR goal=`income` → returns `income-focused`. These are checked in order (taxAware first, then intl, then income), so **taxAware beats intl beats income** when multiple are selected. Like A1, the computed risk score is ignored—each variant is a single fixed portfolio.

#### A3. Investment Style Override
- **Source**: `matchingEngine.js` lines 195–202
- **Inputs**: `investment-style` step (`index`, `blend`, `active`)
- **Effect**: If style=`index` → returns `all-index-core` (risk 6). If style=`active` → returns `active-alpha` (risk 7). Style=`blend` does **not** trigger an override; it falls through to core matching. Again, the computed risk score is ignored.

### Tier B — Base Risk Score (largest single input to scoring)

#### B1. Volatility Reaction (the "risk" question)
- **Source**: `questions.js` lines 71–81, consumed at `scoringUtils.js` line 12
- **Question**: "If your portfolio dropped 20% in a month, what would you do?"
- **Answers and base scores**:
  - "Sell everything" → riskScore: **2**
  - "Sell some to limit losses" → riskScore: **4**
  - "Hold steady and wait it out" → riskScore: **6**
  - "Buy more at lower prices" → riskScore: **8**
  - "Significantly increase my position" → riskScore: **10**
- **Impact**: Sets the base score that all modifiers adjust. A swing from "Sell everything" (2) to "Significantly increase" (10) is an 8-point range — the single largest determinant when no override is active.
- **Default**: If unanswered, defaults to **3** (`scoringUtils.js` line 12: `answers.risk?.riskScore || 3`)

### Tier C — Major Modifiers (±2 to ±3 aggregate)

#### C1. Growth-vs-Stability Preference
- **Source**: `questions.js` lines 17–28, consumed at `scoringUtils.js` lines 33–37
- **Question**: "Calibrate growth vs. stability — where do you fall on the spectrum?"
- **Answers and nudges**:
  - "1 — Preserve" → riskNudge: **-2**
  - "2 — Conservative" → riskNudge: **-1**
  - "3 — Balanced" → riskNudge: **0**
  - "4 — Growth" → riskNudge: **+1**
  - "5 — Max Growth" → riskNudge: **+2**
- **Impact**: Soft modifier, ±2 max. Applied additively to base score.

#### C2. AI Behavioral Insights (two questions combined)
- **Source**: `scoringUtils.js` lines 45–52; AI analysis prompt in `api/ai-insight.js`
- **Questions**: Two open-ended questions generated by GPT-4o-mini. The first probes emotional/behavioral relationship with money. The second pressure-tests the profile for contradictions.
- **Answers**: Free-text, analyzed by the AI. Returns `riskModifier` clamped to **[-1.5, +1.5]** per question.
- **Impact**: Combined impact up to **±3.0** across both insights. Also produces `suggestedEmphasis` (growth/stability/income/balanced) used as a tiebreaker in Tier 4 matching.
- **Fallback**: If the API fails, returns `riskModifier: 0, suggestedEmphasis: 'balanced'` (neutral, no impact).

#### C3. Deep-Dive Questions (6 sub-questions)
- **Source**: `questions.js` lines 121–187, consumed at `scoringUtils.js` lines 16–25
- **Gate**: User must opt in at the `deep-dive-prompt` step; if declined, these are skipped entirely.
- **Sub-questions and modifier ranges**:

| Sub-question | Options | Modifier range |
|---|---|---|
| Savings percentage | Under 10% (0), 10–25% (0), 25–50% (-0.25), Over 50% (-0.5) | 0 to -0.5 |
| Investing experience | Never (-0.5), <1yr (-0.25), 1–5yr (0), 5+yr (+0.5) | -0.5 to +0.5 |
| Emergency fund | 6+mo (+1), 3–6mo (+0.5), Building (0), No (-1) | -1.0 to +1.0 |
| Income stability | Very stable (+0.5), Stable (0), Variable (-0.5), Uncertain (-1) | -1.0 to +0.5 |
| Withdrawal likelihood | Very unlikely (+0.5), Unlikely (0), Possible (-0.5), Likely (-1) | -1.0 to +0.5 |
| Checking behavior | Quarterly (+1), Monthly (+0.5), Weekly (0), Daily (-0.5), Immediately (-1) | -1.0 to +1.0 |

- **Aggregate impact**: Theoretical range **-5.0 to +4.0**, but realistic range is roughly **-3.0 to +3.0** since extreme answers on every question simultaneously is unlikely.

### Tier D — Minor Modifiers (±1 to ±1.5)

#### D1. Goal Follow-Up Risk Nudge
- **Source**: `scoringUtils.js` lines 27–31; follow-up definitions in `questions.js` GOAL_FOLLOWUPS
- **Questions**: Vary by goal. For retirement: age bracket. For education: years until needed. For income: when income should begin. For emergency: months of expenses targeted. For home: target savings amount.
- **Impact**: Most follow-ups provide `impliedTimeline` (see D3 below) but only some provide `riskNudge`. Looking at the code, the retirement and income follow-ups in GOAL_FOLLOWUPS have `impliedTimeline` but **no explicit riskNudge**. The home follow-up has neither riskNudge nor impliedTimeline. This modifier fires only if `followup?.riskNudge` is truthy — meaning **for the current follow-up definitions, this modifier is effectively unused** (see Gaps section).

#### D2. Goal-Conditional Risk Nudge
- **Source**: `scoringUtils.js` lines 39–43; conditionals in `questions.js` GOAL_CONDITIONALS
- **Questions**: Only appear for specific goal+age combinations:
  - `retirement-older` (retirement goal, age ≥50): "Do you have other retirement savings?" → Significant savings: riskNudge **+1**, None/primary: riskNudge **-1**, Some: no nudge
  - `home`: "First time saving for a major goal?" → No riskNudge on either answer
  - `education`: "Considering a 529 plan?" → No riskNudge on any answer
- **Impact**: Only the retirement-older conditional actually carries a riskNudge (±1). The home and education conditionals are informational only.

#### D3. Timeline Uplift
- **Source**: `scoringUtils.js` lines 3–9 and 56–58
- **Input**: Effective timeline (user's explicit timeline selection, or impliedTimeline from goal follow-up)
- **Values**: `20+` → **+1.0**, `10-20` → **+0.5**, all others → **0**
- **Impact**: Always positive, rewards long horizons. Maximum +1.0.

### Tier E — Post-Selection Modifiers (don't affect portfolio choice)

#### E1. Theme Overlays
- **Source**: `matchingEngine.js` lines 4–48; `portfolios.js` THEME_OVERLAYS
- **Question**: "Are you interested in any investment themes?" (multi-select)
- **Options**: AI (BAI), Clean Energy (POWR), Innovation (DYNF), Dividend (DGRO), None
- **Impact**: Applied **after** portfolio selection. Injects thematic ETFs into the chosen portfolio's holdings. Does not change which portfolio is selected.

### Tier F — Captured but Unused in Scoring

#### F1. Financial Picture (sliders)
- **Source**: `questions.js` line 31–35 (type: `financial-picture`)
- **Inputs**: Current age, retirement age, current savings, annual income, savings rate
- **Impact on scoring**: **None directly**. The `currentAge` value is used in `useQuestionnaire.js` (line 13) to determine whether to show the `retirement-older` conditional, but the sliders themselves contribute no modifier to `computeRiskScore`. They appear to be used for the live projection visualization only.

#### F2. Account Type
- **Source**: `questions.js` lines 44–54
- **Inputs**: Taxable, Traditional IRA, Roth IRA, 401(k), Unsure
- **Impact on scoring**: **None**. No riskNudge, no modifier, not referenced in `computeRiskScore` or `matchPortfolio`. Used only to determine which goal-conditional to show next (structurally, via step ordering).

## 2. Answer-to-Portfolio Mapping — Worked Examples

### Example A: Conservative Retiree
**Profile**: Goal=Retirement, Age=62 (60+), Risk reaction="Sell some" (4), Growth pref=2 (Conservative), Timeline=2-5yr (from follow-up impliedTimeline), AI insights both neutral, deep-dive declined, no preferences, style=blend, no themes.

**Scoring** (`scoringUtils.js`):
- Base score: **4** (sell-some)
- Risk-preference nudge: **-1** (conservative)
- Timeline uplift: **0** (2-5 years)
- AI modifiers: **0** (neutral)
- Deep-dive: skipped
- **Final: round(4 + (-1) + 0) = 3**

**Matching** (`matchingEngine.js`):
1. Near-term trigger? Goal=retirement is NOT in {home, education, emergency} → **skip**
2. Preferences? None toggled → **skip**
3. Style? blend → **skip** (only index/active trigger)
4. Core matching:
   - Candidates: all 10 core portfolios (no preference/style)
   - Goal filter: retirement is in suitableFor for portfolios 2–9 (conservative through aggressive) → 8 candidates
   - Timeline filter: 2-5 must be ≥ each portfolio's minTimeline. Using `timelineAtLeast('2-5', minTimeline)`: ultra-conservative (under-2 ✓), conservative (under-2 ✓), mod-conservative (2-5 ✓), conservative-balanced (2-5 ✓). Portfolios requiring 5-10 or longer are filtered OUT.
   - Intersection of goal + timeline filters: **conservative (2), mod-conservative (3), conservative-balanced (4)**
   - Distance to score 3: conservative=|2-3|=1, mod-conservative=|3-3|=0, conservative-balanced=|4-3|=1
   - **Winner: Moderately Conservative Portfolio (risk 3)**

**Result**: Moderately Conservative Portfolio, 4–6% target return, bond-heavy with equity component.

### Example B: Young Aggressive Wealth Builder
**Profile**: Goal=Wealth Building, Timeline=20+yr, Risk reaction="Significantly increase" (10), Growth pref=5 (Max Growth, +2), Deep-dive: 5+ years experience (+0.5), 6+ month emergency fund (+1), very stable income (+0.5), very unlikely withdrawal (+0.5), quarterly checking (+1), under-10% of savings (0). AI insight 1: riskModifier +1.0. AI insight 2: riskModifier +0.5, suggestedEmphasis='growth'. Style=blend, themes=AI+Innovation.

**Scoring**:
- Base: **10**
- Risk-preference: **+2**
- Deep-dive total: 0 + 0.5 + 1 + 0.5 + 0.5 + 1 = **+3.5**
- AI insight 1: **+1.0**
- AI insight 2: **+0.5**
- Timeline uplift: **+1.0** (20+ years)
- Raw total: 10 + 2 + 3.5 + 1.0 + 0.5 + 1.0 = **18.0**
- **Clamped to 10** (max)

**Matching**:
1. Near-term trigger? wealth-building not in {home, education, emergency} → skip
2. Preferences? None → skip
3. Style? blend → skip
4. Core matching:
   - Goal filter: wealth-building in suitableFor for conservative-balanced(4), balanced(5), balanced-growth(6), growth(7), aggressive-growth(8), aggressive(9), max-growth(10)
   - Timeline filter: 20+ ≥ all minTimelines → all 7 pass
   - Distance to score 10: max-growth(10)=0
   - **Winner: Maximum Growth Portfolio (risk 10)**

**Post-selection**: AI theme (BAI, 10%) and Innovation theme (DYNF, 10%) overlaid. DYNF already exists in max-growth at 17% — boosted to 22% (min(17+5, 25)). BAI added as new holding, largest holding (IVV at 20%) reduced to max(20-10, 5) = 10%.

### Example D: Preference Override Trumps Score
**Profile**: Goal=Retirement, Age=35, Timeline=20+, Risk reaction="Hold steady" (6), Growth pref=3 (Balanced), AI insights neutral, deep-dive declined. Preferences: taxAware=true, intl=true. Style=index.

**Scoring**:

## 3. Override vs. Score-Based Decision Points

### Hard Override Paths (complete list)

| Priority | Condition | Resulting Portfolio | Risk Score Used? |
|---|---|---|---|
| 1 | Goal ∈ {home, education, emergency} AND timeline='under-2' | `near-term-reserve` | No |
| 2a | preferences.taxAware = true | `tax-aware-balanced` | No |
| 2b | preferences.intl = true | `global-explorer` | No |
| 2c | preferences.income = true OR goal='income' | `income-focused` | No |
| 3a | investment-style = 'index' | `all-index-core` | No |
| 3b | investment-style = 'active' | `active-alpha` | No |

Overrides are evaluated top-to-bottom. The first match wins. A user who selects taxAware + intl + income + active style will always get `tax-aware-balanced` regardless of the other selections.

### Score-Based Matching (when no override fires)

This path executes when: goal is NOT near-term-triggering with under-2 timeline, no preference toggles are active, goal is not 'income', and style is 'blend' or unanswered.

**Score-to-Portfolio Mapping Table** (`portfolios.js`):

| Risk Score | Portfolio | Target Return | Suitable Goals | Min Timeline |
|---|---|---|---|---|
| 1 | Ultra-Conservative | 2–3% | home, income, emergency | under-2 |
| 2 | Conservative | 3–5% | home, retirement, income, emergency | under-2 |
| 3 | Mod-Conservative | 4–6% | home, retirement, income, education, emergency | 2-5 |
| 4 | Conservative Balanced | 5–7% | retirement, home, income, wealth-building, education, emergency | 2-5 |
| 5 | Balanced | 6–8% | retirement, wealth-building, education, home, emergency | 5-10 |
| 6 | Balanced Growth | 6–9% | retirement, wealth-building, education | 5-10 |
| 7 | Growth | 7–10% | retirement, wealth-building, education | 10-20 |
| 8 | Aggressive Growth | 8–11% | retirement, wealth-building | 10-20 |
| 9 | Aggressive | 9–12% | retirement, wealth-building | 20+ |
| 10 | Maximum Growth | 10–13% | retirement, wealth-building | 20+ |

**Filtering logic** (`matchingEngine.js` lines 204–218):
1. Start with all 10 core portfolios
2. Filter by goal compatibility (`suitableFor.goals` must include user's goal). If filtering leaves 0 results, the filter is skipped (fail-open).
3. Filter by timeline compatibility (`timelineAtLeast(userTimeline, portfolio.minTimeline)`). If filtering leaves 0 results, the filter is skipped (fail-open).
4. Sort remaining by distance to computed risk score (ascending)

**Tie-breaking** (`matchingEngine.js` lines 220–233):
1. Primary: smallest absolute distance to risk score
2. Secondary: AI `suggestedEmphasis` tiebreaker — if emphasis is 'growth', prefer portfolios with riskScore ≥ 7; if 'stability', prefer ≤ 4; if 'income', prefer income-preference portfolios (note: this can never match in core-only pool since income-preference portfolios were excluded)
3. Tertiary: higher riskScore wins (growth bias) — `b.riskScore - a.riskScore`

## 4. Gaps and Dead Ends

### 4.1 Inputs Captured but Unused in Matching

**Financial Picture sliders** (`questions.js` line 31–35): The user enters current age, retirement age, savings, income, and savings rate. These are **never consumed** by `computeRiskScore` or `matchPortfolio`. The only downstream use is: `currentAge` (from sliders) helps determine whether the `retirement-older` conditional appears (`useQuestionnaire.js` line 13). The sliders power a live projection visualization, but the projection doesn't feed back into scoring.

**Account type** (`questions.js` lines 44–54): Captured but never referenced in scoring or matching. Not even used to influence the tax-aware preference. A user selecting "Taxable brokerage" doesn't automatically get tax-aware positioning — they have to manually toggle it on the preferences step.

**Goal follow-up for home** (`questions.js` lines 234–243): The home follow-up asks about target savings amount ($50K–$200K+) but provides **neither `impliedTimeline` nor `riskNudge`**. The answer is captured in `answers['goal-followup']` but its value never influences any downstream decision. Compare with retirement and education follow-ups which at least provide `impliedTimeline`.

**Goal-conditional for home and education**: The home conditional ("First time saving for a major goal?") and education conditional ("Considering a 529 plan?") have no `riskNudge` on any option. They are pure UI — the answers are stored but never consumed.

**Timeline confidence from AI** (`api/ai-insight.js`): The AI analysis returns `timelineConfidence: 'short' | 'medium' | 'long'`, but this field is **never read** by `computeRiskScore` or `matchPortfolio`. Only `riskModifier` and `suggestedEmphasis` are consumed.

### 4.2 Score Modifiers That Are Defined but Not Wired In

**Goal follow-up riskNudge path exists but no data feeds it**: `scoringUtils.js` lines 27–31 check `answers['goal-followup']?.riskNudge`. However, looking at every option across all GOAL_FOLLOWUPS in `questions.js`, **none of the options define a `riskNudge` property** — they only define `impliedTimeline` (or nothing, in the case of home). The code path is active but the data never triggers it.

### 4.3 Suitability Filters That Could Produce Surprising Results

**Goal=income + no preferences override**: If a user selects goal=`income` but does NOT toggle the `income` preference, and style is `blend`, the income match at line 192 (`preferences.income || goal === 'income'`) will catch it and return `income-focused`. This is actually correct behavior — but note that it means the income preference toggle is redundant for income-goal users.

**Goal filter fail-open behavior** (`matchingEngine.js` lines 209–212): If the goal filter eliminates all candidates, the filter is skipped entirely and all 10 portfolios remain eligible. This means a goal like `income` — which is only in suitableFor for portfolios 1–4 — would normally restrict choices to conservative portfolios. But if some OTHER filter (timeline) had already narrowed the pool such that no remaining portfolios support `income`, the goal filter silently becomes a no-op. In practice this is hard to trigger because the income goal reaches the income-focused override (line 192) before core matching.

**Wealth-building goal, under-2 timeline**: A user with goal=`wealth-building` and timeline=`under-2` avoids the near-term override (wealth-building is not in the trigger set). In core matching, the timeline filter allows portfolios with `minTimeline: 'under-2'` — which are ultra-conservative (1) and conservative (2). The goal filter allows wealth-building only for portfolios 4–10. The intersection is **empty**. Both filters fail-open independently, so the system falls back to: first apply goal filter (portfolios 4–10), then apply timeline filter to those — but `timelineAtLeast('under-2', '2-5')` returns false for portfolios 4+, so the timeline filter eliminates all. Timeline filter then fails-open, restoring all goal-filtered candidates (4–10). The user gets matched to the closest-score portfolio among 4–10, ignoring timeline entirely. A short-term wealth builder could land in the Maximum Growth portfolio.

**Aggressive risk + short timeline + conservative goal**: A user with goal=`emergency`, timeline=`2-5`, risk="Significantly increase" (base 10), growth pref 5 (+2) could get a final score of 12 → clamped to 10. But the near-term override doesn't trigger (timeline is 2-5, not under-2), and no preferences are active. In core matching: emergency goal is in suitableFor for portfolios 1–5, and timeline 2-5 allows minTimeline ≤ 2-5 (portfolios 1–4). Intersection: portfolios 1–4. Closest to score 10 is conservative-balanced (4), distance=6. The user gets Conservative Balanced despite extreme risk tolerance — the goal+timeline suitability filters dominate.

### 4.4 Preference Priority Could Surprise Users

The preference checks at lines 182–193 are mutually exclusive by position, not by design. A user who carefully toggles all three preferences (taxAware + intl + income) will get the tax-aware portfolio and see their international and income preferences silently ignored. The UI doesn't communicate this priority ordering.

### 4.5 Style Override Ignores Risk Score

The `all-index-core` portfolio is hardcoded at risk score 6 and `active-alpha` at risk score 7. A user with a calculated risk score of 2 (very conservative) who selects "Pure index investing" will get a risk-6 portfolio — significantly more aggressive than their profile warrants. The style variants are not risk-tiered.

### 4.6 Holdings Mutation Bug Risk in Theme Overlays

`applyThemeOverlays` (`matchingEngine.js` line 14) clones the holdings array with `[...portfolio.holdings]`, but this is a **shallow clone** — the individual holding objects `{ticker, weight}` are still shared references. When the function mutates `existing.weight` (line 21) or `largest.weight` (line 25), it modifies the original portfolio definition in the `PORTFOLIOS` array. On the first call this works fine, but if `matchPortfolio` were ever called twice in the same JS runtime without a page refresh, the second call would operate on already-mutated weights. In a React app with hot module replacement or strict mode double-rendering, this could produce incorrect allocations.

### 4.7 Truncated Source Files on Disk

All four core pipeline files (`scoringUtils.js`, `matchingEngine.js`, `portfolios.js`, `questions.js`) are **truncated on the filesystem** — each file on disk is significantly smaller than its committed version in git HEAD. For example, `scoringUtils.js` is 1,226 bytes on disk vs. 2,253 bytes in git; `matchingEngine.js` is 3,372 vs. 9,335 bytes. The files end mid-token (e.g., `scoringUtils.js` ends at `analysis?.ri`). The app cannot function from the current working tree state — it would only work from a clean git checkout. This appears to be a sync or write issue rather than intentional.
